前后端分离经验

API是什么

在前后端开发中，API（Application Programming Interface，应用程序编程接口）指的是：

➡️ 前端和后端之间的“通信接口”

简单来说，API 就是一套规定好的请求方式和数据格式，让前端可以向后端“要数据”或“提交数据”。

例如：

  • 前端想获取用户列表 → 调后端的 /api/users

  • 前端提交登录信息 → 调后端的 /api/login

  • 前端发送一个 POST 请求 → 后端根据 API 规范处理这个请求并返回结果

API 的作用

  1. 前后端解耦
    前端不需要知道后端内部怎么实现，只需要按 API 格式发送请求。

  2. 数据传输标准化
    通常用 JSON 作为数据格式，统一规范。

  3. 清晰的开发协作方式
    后端制定 API 文档，前端根据文档开发，双方可以并行工作。

API 一般包含内容：

  • 请求方式（GET / POST / PUT / DELETE）

  • 请求地址（URL）

  • 请求参数（query、body）

  • 返回结果格式（JSON）

  • 错误代码说明

举个例子（典型 REST API）

REST API（Representational State Transfer API）是一种后端提供 API 的设计风格（不是框架，不是技术），它使用类似HTTP的风格表达对后端数据的请求。

前端请求：GET /api/user/123

后端返回：{ "id": 123, "name": "Alice", "age": 25 }

URL是资源定位符，也就是我们常说的网址，上面前端的网址是缺省的，会被填入当前所处的非完全或配置的后端服务器地址作为前缀，/api是约定俗成的对数据请求的地址，/user/123表示前端希望访问user中id为123的数据。

对这个URL传入GET指令之后，后端返回了一段由key-value对组成的json数据，前端就可以从中读取需要的数据。

HTTP和TCP

如果你使用 TCP (Qt C++ QTcpSocket / QTcpServer)

这是我上一条回复给你的代码方案。

  • 痛点（粘包问题）：

    • TCP 是流式的。如果你连续快速发送两个 JSON：{"id":1} 和 {"id":2}。

    • 服务器可能会一次性收到：{"id":1}{"id":2}。

    • 你必须写的代码：你需要自己设计一个协议，比如在 JSON 前面加 4 个字节表示长度。如果不写这个，程序不稳定，数据量一大就崩溃。

  • 优势：

    • 实时性：如果航班价格突然变动，服务器可以直接通过 Socket 主动推送给客户端，不需要客户端一直问。

如果你使用 HTTP (Qt C++ QNetworkAccessManager)

这是更接近 Web 开发的方案。

  • 优势：

    • 不用管粘包：HTTP 协议头里有 Content-Length，Qt 的 HTTP 类库会自动帮你收完数据才通知你。

    • 代码清晰：你只关注业务逻辑（GET 获取航班，POST 添加航班）。

    • 生态好：如果你以后想写个 Python 脚本或者用 手机 App 访问你的数据库，HTTP 接口是通用的，而自定义的 TCP 协议只有你的 Qt 客户端能懂。

  • 劣势（在 C++ 中）：

    • 后端难写：Qt 默认的 QTcpServer 很底层。要用 Qt 写一个好用的 HTTP Server (后端)，你需要引入第三方库（如 QtWebApp 或 Drogon），或者使用较新的 QHttpServer (Qt 6.4+)。

对你的建议：怎么选？

鉴于你是软件工程专业的学生，且项目是航班管理系统：

方案 A：坚持使用 TCP (推荐用于学习底层原理)

如果你想深入理解网络编程的本质，弄懂什么是 Socket、什么是字节流、如何处理并发，继续使用 TCP。

  • 挑战：你需要给上一条代码加上“包头长度封装”，解决粘包问题。

  • 加分项：在答辩或面试时，你能说出“我自己设计了应用层协议来解决 TCP 粘包”，这是一个亮点。

方案 B：使用 HTTP (推荐用于追求开发效率和架构现代感)

如果你更关注业务逻辑（数据库增删改查），或者想让你的架构更像现在的互联网大厂架构（RESTful API）。

  • 做法：

    • 后端：建议改用 Qt 6 的 QHttpServer 模块，或者干脆用 Python (Flask/FastAPI) / Go / C# 做后端（C++写后端Web接口稍微有点累）。

    • 前端：Qt 的 QNetworkAccessManager 发送 GET/POST 请求。

  • 优点：你可以用 Postman 测试接口，开发速度快，Debug 容易。

总结

  • TCP = 自己造车。硬核，快，难控制，适合游戏、实时控制、高频交易。

  • HTTP = 坐出租车。方便，标准，稍微慢一点，适合网站、管理系统、App 接口。

目前的航班管理系统属于“管理系统”，标准做法其实是 HTTP。但如果你是为了完成 C++ 课设/校赛，用 TCP 会显得你的 C++ 功底更扎实。

函数示例：获取航班信息

第一阶段：发起网络请求

  • 目的：告诉服务器“我要航班列表”。

  • QUrl：指定目标地址。127.0.0.1 代表本机，8080 是端口，/api/flights 是接口路径。

  • manager->get(request)：使用 QNetworkAccessManager 的一个对象manager发起一个 HTTP GET 请求。这是一个异步操作，代码不会在这里卡住等待，而是继续向下执行。

  • QNetworkReply类指针reply用来储存这次请求的结果。

第二阶段：异步处理响应

  • 核心机制：信号与槽（Signals & Slots）。

  • 逻辑：当监控到reply指向的对象发出finished信号，且this指针指向的对象没有被销毁时时，Qt 会自动调用后面的Lambda函数。

  • [=]：这是 C++ Lambda 的捕获列表，表示将外面作用域的局部变量（如 this 指针）按值捕获进来，捕获this之后在 Lambda 内部就可以访问 tableWidget 等成员变量，编译器会自动翻译成this->tableWidget。

第三阶段：JSON 数据解析（代码中最关键的逻辑）

  • 鲁棒性设计：这里做了一个很棒的兼容性处理。后端返回的 JSON 格式通常有两种流派，这段代码都能处理：

    1. 纯数组流：[ {"flight": "CA101"}, ... ] -> 直接转为数组。

    2. 对象包裹流：{ "code": 200, "msg": "success", "data": [ ... ] } -> 提取 data 字段里的数组。
      第四阶段：更新 UI (QTableWidget)

  • 流程：先清空表格，然后循环遍历 JSON 数组，将每个字段（如 flight_number）填入对应的单元格。

  • 内存管理：最后调用 reply->deleteLater()，告诉 Qt 在适当的时候释放 reply 对象占用的内存，防止内存泄漏。
